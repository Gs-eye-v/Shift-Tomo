# カレンダーアプリ プロジェクト定義書 (CONTEXT.md)

## 【厳格な開発ルール】
あなたはシニア・Flutterアーキテクトです。コードのスパゲッティ化を防ぐため、以下のルールを完全に遵守してください。破りそうになった場合はコードを書かずに相談すること。

1. **アーキテクチャの強制（Feature-first）**
   - ディレクトリ構成は「Feature-first」を採用。`lib/src/features/{機能名}/` の下に `ui`, `provider`, `model`, `repository` 等を作成しカプセル化する。
   - 共通パーツは `lib/src/common_widgets/` や `lib/src/utils/` に切り出す。
2. **状態管理（Riverpod）のルール**
   - 最新の `riverpod_generator`（`@riverpod`）を使用。古い構文（StateProvider等）は使用禁止。
   - UI（Widget）内にビジネスロジックを書かない。
   - 非同期処理は `AsyncNotifier` に記述し、UIでは `AsyncValue` の `.when()` でローディング/エラーをハンドリングする。
   - UIからの直接的な状態書き換えは禁止。必ずNotifier内のメソッドを呼ぶ。
3. **AIの認知限界への対策（300行ルール）**
   - 1ファイルは300行（コメント含む）以内。超える場合はWidgetやロジック（純粋なDartクラス）を別ファイルに分割する。
4. **安全な進行（1タスク・1コミット）**
   - 複数機能を同時に実装しない。指示された機能のみを最小限のコードで実装する。
   - 新機能開発と既存コードのリファクタリングを混ぜない。
5. **外部記憶の維持（このファイルの更新）**
   - 大きな機能実装が終わるたびに、あなたがこの `CONTEXT.md` の「現在の実装状況・次の課題」を更新すること。

## 【アプリの基本仕様】
- **コンセプト:** 完全プライバシー保護と予定共有を両立する勤務・シフト管理カレンダー
- **技術スタック:** Flutter, Riverpod, SQLite (ローカルDB), Supabase (暗号化同期用)
- **コアバリュー:** 視覚的な分かりやすさ（色＋1文字＋絵文字）、ローカルファースト、E2EE暗号化通信。

## 【UI/UX 設計仕様】
- **テーマ:** 「ダーク」「ライト」「ポップ」「ライン風」の4種類を実装（`ThemeData`で切り替え）。デフォルトはダーク（Deep Gray #121212）。
- **ヘッダー領域:**
  - ユーザー切り替えタブ（横スクロールのアバター/名前リスト）。左端は「自分」。
  - 右端に「+」ボタン（同期ID入力、または新規ローカルユーザー作成）。
- **カレンダーマス目 (Cell) の厳格な配置ルール（4分割レイアウト）:**
  - **背景色:** シフト定義色（早番＝薄オレンジ等）。1日に複数シフトがある場合は**縦に等分して色を塗る**。
  - **透かし文字:** 中央に大きく薄く「1文字（早、休など）」を配置。
  - **左上:** 勤務体系の絵文字（🐤早番, 🐔通常, 🍗遅番, 🥚休み 等）
  - **右上:** サブタスク1の絵文字（💪ジム 等）
  - **左下:** サブタスク2の絵文字
  - **右下:** メモありフラグ（📝）
  - ※ 勤務と休みの絵文字は左上を最優先とする。

## 【データベース・同期仕様】
- **ローカルDB (SQLite):** `profiles`, `shift_tags`, `shifts` テーブルを使用し、生データはすべてデバイス内に保存。
- **同期機能 (Supabase E2EE):**
  - クラウドには個人情報を上げない。
  - ルームIDとパスワードによるE2EE（エンドツーエンド暗号化）を採用。暗号化された文字列のみをSupabaseへ送信・保存する。
  - 他人のデータは「閲覧専用」として読み込む（共同編集によるコンフリクトを回避）。
